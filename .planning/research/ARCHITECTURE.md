## Token Architecture: Spec → Config → Component

The token flow is a strict one-way pipeline. Each layer depends only on the layer above it.

```
morph-maternity-design-system.html  (brand spec — source of truth)
        |
        v
tokens/*.ts                          (raw token values as TypeScript constants)
        |
        v
tailwind.config.ts                   (extends Tailwind theme with token values)
        |
        v
NativeWind JSX transform             (converts className → StyleSheet at build time)
        |
        v
Component className="..."            (components reference only Tailwind class names)
        |
        v
Storybook / Momcare app              (consumers render components)
```

Each token file exports typed constants. These are plain TypeScript — no framework dependency.

### Layer 2: Tailwind Config (`tailwind.config.ts`)

The config imports token values and extends the Tailwind theme. This is the only file that bridges tokens into class names.

```typescript
// tailwind.config.ts
import type { Config } from 'tailwindcss';
import { colors, fontFamilies, fontSizes, spacing, borderRadius, shadows } from './tokens';

const config: Config = {
  content: [
    './src/**/*.{ts,tsx}',
    './.storybook/**/*.{ts,tsx}',
  ],
  presets: [require('nativewind/preset')],
  theme: {
    extend: {
      colors: {
        rose:    colors.rose,
        cream:   colors.cream,
        sage:    colors.sage,
        neutral: colors.neutral,
        // Semantic aliases — map to palette
        primary:    colors.rose[500],
        'primary-dark': colors.rose[600],
        surface:    colors.cream[50],
        background: colors.cream[100],
        'on-surface': colors.neutral[900],
        muted:      colors.neutral[400],
        border:     colors.neutral[200],
        error:      '#D32F2F',
        success:    '#388E3C',
        warning:    '#F57C00',
      },
      fontFamily: {
        display: [fontFamilies.display],
        body:    [fontFamilies.body],
        mono:    [fontFamilies.mono],
      },
      fontSize:     fontSizes,
      spacing:      spacing,
      borderRadius: borderRadius,
      boxShadow:    shadows,
    },
  },
  plugins: [],
};

export default config;
```

**Critical NativeWind v4 note:** NativeWind v4 uses a `preset` rather than the v2 plugin pattern. The `nativewind/preset` must be listed under `presets`, not `plugins`. This is a breaking change from v2/v3.

### Layer 3: Component Usage

Components reference only Tailwind class names. They never import from `tokens/` directly.

```typescript
// src/components/actions/PrimaryButton/Button.tsx
import { Pressable, Text } from 'react-native';

interface PrimaryButtonProps {
  label: string;
  onPress: () => void;
  disabled?: boolean;
  size?: 'sm' | 'md' | 'lg';
}

const sizeClasses = {
  sm: 'px-4 py-2 text-sm',
  md: 'px-6 py-3 text-base',
  lg: 'px-8 py-4 text-lg',
};

export function PrimaryButton({ label, onPress, disabled, size = 'md' }: PrimaryButtonProps) {
  return (
    <Pressable
      onPress={onPress}
      disabled={disabled}
      className={`
        bg-primary rounded-lg items-center justify-center
        active:bg-primary-dark
        disabled:opacity-50 disabled:cursor-not-allowed
        ${sizeClasses[size]}
      `}
    >
      <Text className="font-body font-semibold text-white">
        {label}
      </Text>
    </Pressable>
  );
}
```

### The `cn()` Utility

All components should use a `cn()` helper for conditional/merged class names. This prevents className conflicts and handles variant merging correctly.

```typescript
// src/utils/cn.ts
import { clsx, type ClassValue } from 'clsx';
import { twMerge } from 'tailwind-merge';

export function cn(...inputs: ClassValue[]): string {
  return twMerge(clsx(inputs));
}
```

Usage in components:

```typescript
className={cn(
  'bg-primary rounded-lg items-center',
  disabled && 'opacity-50',
  size === 'lg' && 'px-8 py-4',
  className  // allow caller to extend/override
)}
```

## Component Composition Patterns

### Pattern 1: Primitive → Composite Composition

Primitives are unstyled or minimally styled base components. Composite components assemble primitives.

```
Text (primitive)
  └── FormSectionHeader (composite — styled Text + optional icon)

Pressable (RN core) + Text (primitive)
  └── PrimaryButton (composite)
  └── OutlineButton (composite — same structure, different className)
  └── IconTextButton (composite — adds Icon primitive)

TextInput (primitive)
  └── PasswordInput (composite — TextInput + visibility toggle Icon)
  └── SearchBar (composite — TextInput + search Icon prefix)
  └── DatePickerInput (composite — TextInput + calendar Icon + picker logic)
```

**Rule:** Primitive components accept a `className` prop for extension. Composite components do not expose raw `className` — they expose semantic props (`variant`, `size`, `state`).

### Pattern 2: Variant Props over Duplicate Components

Use a `variant` prop instead of separate `PrimaryButton` / `SecondaryButton` components where the structure is identical.

```typescript
// Prefer this:
<Button variant="primary" label="Submit" onPress={...} />
<Button variant="outline" label="Cancel" onPress={...} />

// Over this:
<PrimaryButton label="Submit" onPress={...} />
<OutlineButton label="Cancel" onPress={...} />
```

Exception: When components diverge structurally (e.g., FAB vs standard button), keep them as separate components.

### Pattern 3: forwardRef for Composability

Form components must use `forwardRef` to allow parent forms to control focus imperatively.

```typescript
const TextInput = forwardRef<RNTextInput, TextInputProps>((props, ref) => {
  return <RNTextInput ref={ref} ... />;
});
TextInput.displayName = 'TextInput';
```

### Pattern 4: Slot/Children for Layout Components

Layout components (MobileShell, ScrollableContent) use typed children slots rather than a flat `children` prop where structure matters.

```typescript
interface MobileShellProps {
  header?: React.ReactNode;   // TopAppBar goes here
  children: React.ReactNode;  // main content
  footer?: React.ReactNode;   // BottomNavBar goes here
}
```

### Pattern 5: Accessible Props as First-Class

Every interactive component must surface `accessibilityLabel`, `accessibilityHint`, and `accessibilityRole` as props — not buried in `...rest`. Healthcare app context makes this non-negotiable.

## Storybook File Organization

**Storybook version:** React Native Storybook v7+ with `@storybook/react-native`.

### Story co-location

Stories live next to their component, not in a separate `__stories__/` folder. This keeps components self-contained and easy to relocate.

```
src/components/actions/PrimaryButton/
  ├── Button.tsx
  ├── Button.types.ts
  ├── Button.stories.tsx    ← co-located story
  └── index.ts
```

### Story file structure

```typescript
// Button.stories.tsx
import type { Meta, StoryObj } from '@storybook/react-native';
import { PrimaryButton } from './Button';

const meta: Meta<typeof PrimaryButton> = {
  title: 'Actions/PrimaryButton',
  component: PrimaryButton,
  argTypes: {
    label:    { control: 'text' },
    disabled: { control: 'boolean' },
    size:     { control: 'select', options: ['sm', 'md', 'lg'] },
  },
  args: {
    label:    'Button Label',
    disabled: false,
    size:     'md',
  },
};

export default meta;
type Story = StoryObj<typeof PrimaryButton>;

export const Default: Story = {};

export const Disabled: Story = {
  args: { disabled: true },
};

export const Large: Story = {
  args: { size: 'lg' },
};
```

```typescript

import React from 'react';
import type { Preview } from '@storybook/react-native';
import '../global.css';  // NativeWind global CSS import

const preview: Preview = {
  decorators: [
    (Story) => (
      <View className="flex-1 bg-surface p-4">
        <Story />
      </View>
    ),
  ],
  parameters: {
    controls: { matchers: { color: /(background|color)$/i } },
  },

```

**NativeWind + Storybook integration note:** NativeWind v4 requires a `global.css` file (with `@tailwind` directives) imported in the Storybook preview. The Metro bundler must be configured with NativeWind's metro plugin to process CSS. On-device Storybook runs inside the Expo dev client — this is the recommended setup over the web target for React Native components.

## Build Order Dependencies

Dependencies flow upward — lower layers must be complete before building upper layers.

```
Layer 0: Project scaffold
  expo init → NativeWind v4 install → tailwind.config.ts → babel.config.js → metro.config.js
  (No component dependencies — must be complete before any component work)

Layer 1: Token system
  tokens/*.ts → tailwind.config.ts extends
  (No component dependencies — must complete before any styled component)

Layer 2: Storybook infrastructure
  .storybook/main.ts → .storybook/preview.tsx → verify on-device rendering
  (Depends on: Layer 0, Layer 1 partial — needs NativeWind working in Storybook)

Layer 3: Primitives
  Text → Icon → Badge
  (Depends on: Layer 1, Layer 2)
  (No inter-component dependencies within this layer)

Layer 4: Form components
  TextInput → PasswordInput, SearchBar, DatePickerInput (all extend TextInput)
  Checkbox, Dropdown, Textarea, FilterChips (independent)
  FormSectionHeader, FormActionRow (compose Text + layout)
  (Depends on: Layer 3 primitives)

Layer 5: Action components
  PrimaryButton, OutlineButton (independent)
  IconTextButton (depends on: Icon primitive)
  FAB (depends on: Icon primitive)
  ToggleSwitch (independent)
  TextLinkPair, OnboardingNavRow (depend on: Text primitive)
  BottomActionBar (composes buttons)
  (Depends on: Layer 3 primitives)

Layer 6: Navigation components
  TopAppBar (composes Icon, Text, PrimaryButton)
  BottomNavBar (composes Icon, Text)
  LanguageToggle (composes ToggleSwitch or custom)
  (Depends on: Layer 3 primitives, Layer 5 partial)

Layer 7: Data display components
  StatCard, DataTable, Timeline, Calendar
  (Depends on: Layer 3 primitives, Layer 4 partial)

Layer 8: Layout / shell components
  MobileShell (wraps TopAppBar + BottomNavBar slots)
  ScrollableContent, SectionDivider, EmptyFooter
  (Depends on: Layer 6 navigation)
```

**Critical dependency:** Icon library must be decided before Layer 4 (FAB, IconTextButton, SearchBar, TopAppBar all embed icons). Deferring icon selection blocks multiple layers.

## Export Pattern

### Per-component barrel (`src/components/[category]/[Component]/index.ts`)

```typescript
// src/components/actions/PrimaryButton/index.ts
export { PrimaryButton } from './Button';
export type { PrimaryButtonProps } from './Button.types';
```

### Category barrel (`src/components/[category]/index.ts`)

```typescript
// src/components/actions/index.ts
export * from './PrimaryButton';
export * from './OutlineButton';
export * from './IconTextButton';
export * from './FAB';
export * from './BottomActionBar';
export * from './ToggleSwitch';
export * from './TextLinkPair';
export * from './OnboardingNavRow';
```

### Root barrel (`src/index.ts`)

```typescript
// src/index.ts — single import surface for consumers
export * from './components/primitives';
export * from './components/forms';
export * from './components/actions';
export * from './components/navigation';
export * from './components/data-display';
export * from './components/layout';
export * from '../tokens';
```

Consumer usage in Momcare app:

```typescript
import { PrimaryButton, TextInput, BottomNavBar } from '../my-design-system/src';
import { colors } from '../my-design-system/tokens';
```

**Named exports only.** No default exports from component files. Default exports break tree-shaking and make refactoring harder (renaming at import site is optional, causing inconsistency).

## Component File Anatomy

Every component folder follows this exact structure. Consistency reduces cognitive overhead.

```
ComponentName/
├── ComponentName.tsx        Main component implementation
├── ComponentName.types.ts   Props interfaces, enums, type helpers
├── ComponentName.stories.tsx Storybook stories (CSF3 format)
└── index.ts                 Re-exports only — no logic
```

**No test files in v1.** Testing scope is deferred; Storybook stories serve as the visual regression baseline.

### ComponentName.types.ts

```typescript
// All props interfaces live here, not inline in the component
export interface PrimaryButtonProps {
  label: string;
  onPress: () => void;
  disabled?: boolean;
  loading?: boolean;
  size?: 'sm' | 'md' | 'lg';
  // Accessibility — always explicit, never via ...rest
  accessibilityLabel?: string;
  accessibilityHint?: string;
}
```

### ComponentName.tsx rules

1. Import from `react-native` — not from `react-native-web` or any web-only lib
2. Use `className` via NativeWind — never `StyleSheet.create()`
3. Import `cn` from `../../utils/cn` for conditional classes
4. No business logic, no API calls, no navigation — pure presentational
5. Export as named export (not default)

## Scalability Considerations

| Concern | v1 (~35 components, 1 consumer) | Future (~80+ components, multiple consumers) |
|---------|----------------------------------|---------------------------------------------|
| Token changes | Edit `tokens/*.ts` + `tailwind.config.ts` | Same — token layer abstracts change |
| Adding dark mode | Add `dark:` variants to className strings | NativeWind v4 supports `dark:` natively |
| Publishing as NPM package | Add `package.json` `main`/`exports` fields | Barrel exports already set up correctly |
| New component category | Add `src/components/[category]/` folder + category barrel | Category barrels keep root barrel stable |
| Performance (Metro bundling) | No concern at 35 components | Tree-shaking via named exports ensures dead code elimination |

## Critical Architecture Constraints

1. **NativeWind v4 preset, not plugin.** `tailwind.config.ts` must use `presets: [require('nativewind/preset')]` — not the v2 `plugins` approach. This is the most common migration mistake.

2. **`global.css` must be imported once.** In Expo, import `global.css` in the root `_layout.tsx` (or equivalent entry). In Storybook, import it in `.storybook/preview.tsx`. Missing this import causes NativeWind styles to silently not apply.

3. **Metro config requires NativeWind plugin.** `metro.config.js` must wrap the default config with `withNativeWind`:

```javascript
// metro.config.js
const { getDefaultConfig } = require('expo/metro-config');
const { withNativeWind } = require('nativewind/metro');

const config = getDefaultConfig(__dirname);
module.exports = withNativeWind(config, { input: './global.css' });
```

4. **Babel config requires NativeWind preset.** Add `nativewind/babel` to Babel presets:

```javascript
// babel.config.js
module.exports = function (api) {
  api.cache(true);
  return {
    presets: [
      ['babel-preset-expo'],
      'nativewind/babel',
    ],
  };
};
```

5. **No cross-layer token imports.** Components never import from `tokens/` directly. Tokens flow through `tailwind.config.ts` only. This ensures that if token values change, only one file needs updating.

6. **Storybook on-device, not web.** The `@storybook/react-native` on-device runner inside Expo Go / Expo Dev Client is the correct target. Web-target Storybook (`@storybook/react-native-web-addon`) can be added later for CI screenshot testing but is not required in v1.
